# 9. Инструменты и технологии

Этот раздел фиксирует не конкретный «обязательный список сервисов», а **классы инструментов**, которые используются в agent-first среде.

Инструменты будут меняться. Поэтому стандарт должен описывать:

1. **какую задачу решает класс инструментов;**
2. **кто должен уметь с ним работать;**
3. **какой уровень зрелости требуется;**
4. **какие ограничения и риски есть.**

---

## 9.1. Главный принцип выбора инструментов

Инструмент выбирается не по модности, а по задаче.

Правильная логика:

> **Задача → данные → риск → нужный результат → инструмент → проверка → внедрение.**

Неправильная логика:

> «Давайте внедрим ChatGPT / агента / n8n / Copilot, а потом придумаем, зачем».

---

## 9.2. Карта классов инструментов

| Класс инструментов          | Для чего нужен                                             | Примеры                                                         |
| --------------------------- | ---------------------------------------------------------- | --------------------------------------------------------------- |
| **Корпоративный AI-чат**    | Безопасная базовая AI-среда для сотрудников                | Внутренний AI-чат, Open WebUI, LibreChat, корпоративный ChatGPT |
| **LLM-ассистенты**          | Тексты, анализ, идеи, документы, таблицы, код, рассуждения | ChatGPT, Claude, Gemini, GigaChat, YandexGPT                    |
| **AI-агенты**               | Выполнение задач по шагам с использованием инструментов    | OpenAI Agents, Claude Code, Copilot Studio agents, Dify agents  |
| **RAG / базы знаний**       | Ответы на основе корпоративных документов и знаний         | Корпоративная база знаний, vector search, Qdrant, OpenSearch    |
| **Research-инструменты**    | Поиск, проверка источников, рыночный анализ                | Deep Research, Perplexity, Gemini, ChatGPT Search               |
| **Документы и офис**        | Тексты, презентации, письма, резюме, протоколы             | Microsoft 365 Copilot, Google Workspace Gemini, Notion AI       |
| **Таблицы и данные**        | Анализ таблиц, выгрузок, расчётов, отклонений              | Excel, Google Sheets, ChatGPT Data Analysis, Power BI           |
| **Workflow automation**     | Связка систем, действий, триггеров и AI-шагов              | n8n, Make, Zapier, Power Automate                               |
| **Agent/workflow builders** | Создание AI-приложений, агентов и сценариев                | Dify, Langflow, Flowise, Copilot Studio                         |
| **Low-code/no-code**        | Быстрые внутренние MVP без тяжёлой разработки              | Baserow, NocoDB, Retool, Appsmith, NocoBase                     |
| **Coding agents**           | Код, скрипты, автоматизации, интеграции                    | Cursor, GitHub Copilot, Kilo Code, Claude Code                  |
| **Дизайн и визуал**         | Макеты, баннеры, презентации, изображения                  | Figma AI, Canva AI, Gamma, Midjourney, Ideogram                 |
| **Контроль и безопасность** | Ограничения, доступы, аудит, проверка данных               | DLP, guardrails, logging, access control, policy checks         |

OpenAI описывает агентов как приложения, которые могут планировать, вызывать инструменты, взаимодействовать со специализированными агентами и сохранять состояние для выполнения многошаговой работы. Microsoft Copilot Studio отдельно описывает автономных агентов, которые реагируют на события, принимают решения и выполняют задачи через triggers, instructions и guardrails. n8n позиционируется как workflow automation tool, который объединяет AI-возможности с бизнес-автоматизацией, а Dify — как платформа для agentic workflows и AI-приложений. ([Разработчики OpenAI][1])

## 9.3. Уровни доступа к инструментам

Инструменты должны выдаваться не хаотично, а по уровню зрелости, роли и рискам.

| Уровень зрелости | Доступные классы инструментов                                               |
| ---------------- | --------------------------------------------------------------------------- |
| **A1**           | Корпоративный AI-чат, готовые шаблоны, базовые LLM-инструменты              |
| **A2**           | Работа с файлами, таблицами, документами, готовыми агентами                 |
| **A3**           | Создание шаблонов, простых AI-сценариев, базовых workflow                   |
| **A4**           | Управление агентами, командными сценариями, no-code/low-code автоматизацией |
| **A5**           | Проектирование agent-first процессов, связка с данными, BI, системами       |
| **A6**           | Портфель AI-инициатив, архитектура применения, трансформация функции        |

---

## 9.4. Корпоративный AI-чат

### Назначение

Базовая безопасная среда для работы сотрудников с ИИ.

Должен использоваться для:

* текстов;
* писем;
* резюме;
* анализа документов;
* подготовки черновиков;
* структурирования задач;
* проверки ошибок;
* объяснения сложных тем;
* работы с внутренними инструкциями, если подключена база знаний.

### Для кого

Для всех сотрудников.

### Минимальный уровень

A1–A2.

### Что важно

Корпоративный AI-чат должен быть предпочтительным инструментом для рабочих задач, особенно если есть риск загрузки внутренних данных.

### Ограничение

Если чат не подключён к актуальным внутренним данным, его ответы не должны восприниматься как источник истины.

---

## 9.5. LLM-ассистенты

### Назначение

Универсальные языковые модели для анализа, текста, логики, структуры, кода, таблиц и рабочих материалов.

Примеры задач:

* подготовить документ;
* сократить текст;
* разложить проблему;
* найти риски;
* сравнить варианты;
* подготовить план;
* сделать протокол;
* проверить логику;
* подготовить письмо;
* сформировать ТЗ.

### Для кого

Для всех сотрудников.

### Минимальный уровень

A1–A2.

### Примеры инструментов

* ChatGPT;
* Claude;
* Gemini;
* GigaChat;
* YandexGPT;
* корпоративные модели;
* локальные модели.

### Важное ограничение

LLM-ассистент не является источником истины. Он может ошибаться, уверенно придумывать факты и неверно трактовать данные. Результат всегда проверяется человеком.

---

## 9.6. AI-агенты

### Назначение

AI-агент используется не для разового ответа, а для выполнения задачи по шагам.

Агент может:

* планировать действия;
* использовать инструменты;
* обращаться к данным;
* работать с файлами;
* вызывать API;
* искать информацию;
* выполнять код;
* передавать задачу другому агенту;
* формировать итоговый результат.

OpenAI Agents SDK поддерживает оркестрацию агентов, включая handoffs между агентами и agents-as-tools для manager-style workflows; Dify описывает Agent node как механизм, где LLM получает автономный контроль над инструментами и сам решает, когда их использовать. ([Разработчики OpenAI][2])

### Для кого

* специалисты — используют готовых агентов;
* старшие/ведущие — создают простые агентные сценарии;
* руководители — внедряют агентов в работу команды;
* ТОПы — перестраивают процессы вокруг агентной модели.

### Минимальный уровень

A2 для использования.
A3–A4 для создания и настройки.
A5–A6 для перестройки процессов.

### Примеры агентов

| Агент                          | Что делает                                 |
| ------------------------------ | ------------------------------------------ |
| **Агент протоколов**           | Делает резюме встреч, решения, поручения   |
| **Агент отчётности**           | Готовит регулярные сводки по данным        |
| **Агент контроля задач**       | Находит просрочки, зависшие задачи, риски  |
| **Агент анализа документов**   | Выделяет риски, расхождения, спорные места |
| **Агент клиентских обращений** | Классифицирует запросы, предлагает ответы  |
| **Агент аналитика**            | Обрабатывает таблицы и формирует выводы    |
| **Агент проверки качества**    | Проверяет материалы по чек-листу           |
| **Coding agent**               | Пишет скрипты, код, автоматизации, тесты   |

### Ограничение

Агент не должен получать право на необратимые действия без контроля человека, если есть риск денег, клиентов, персональных данных, юридических последствий или репутационного ущерба.

## 9.7. RAG и корпоративные базы знаний

### Назначение

RAG нужен, чтобы ИИ отвечал не «из головы модели», а на основе корпоративных документов, инструкций, регламентов, данных и базы знаний.

Используется для:

* поиска по внутренним документам;
* ответов на вопросы сотрудников;
* поддержки адаптации;
* работы с регламентами;
* анализа накопленных знаний;
* подготовки справок;
* поддержки контакт-центра;
* поддержки HR, юристов, коммерции, операций.

### Для кого

Для всех сотрудников как пользователей.
Для руководителей, экспертов, ИТ и владельцев процессов — как инструмент развития знаний.

### Минимальный уровень

A2 для использования.
A4–A5 для проектирования сценариев.

### Примеры технологий

* vector database;
* embeddings;
* Qdrant;
* OpenSearch;
* PostgreSQL + pgvector;
* корпоративные knowledge bases;
* RAG-коннекторы к документам, CRM, таск-трекерам.

### Ограничение

RAG не решает проблему плохой базы знаний. Если документы устарели, противоречат друг другу или плохо структурированы, агент будет воспроизводить хаос.

---

## 9.8. Research-инструменты

### Назначение

Инструменты для поиска, проверки и анализа внешней информации.

Используются для:

* рыночного анализа;
* конкурентного анализа;
* поиска источников;
* первичного исследования;
* проверки фактов;
* подготовки обзоров;
* сбора вариантов;
* анализа трендов.

### Примеры

* Deep Research;
* Perplexity;
* ChatGPT Search;
* Gemini;
* Claude с web search, если доступно;
* специализированные базы данных;
* отраслевые источники.

### Для кого

* ТОПы;
* руководители;
* маркетинг;
* продукт;
* коммерция;
* аналитика;
* стратегия;
* HR;
* закупки;
* развитие.

### Минимальный уровень

A2–A3.

### Ограничение

Research-инструменты не заменяют проверку источников. Вывод должен разделять:

* факт;
* источник;
* интерпретацию;
* гипотезу;
* рекомендацию.

## 9.9. Документы, офис и презентации

### Назначение

Инструменты для ускорения подготовки рабочих материалов.

Используются для:

* писем;
* документов;
* презентаций;
* регламентов;
* протоколов;
* инструкций;
* внутренних коммуникаций;
* управленческих материалов.

### Примеры

* Microsoft 365 Copilot;
* Google Workspace Gemini;
* Notion AI;
* Gamma;
* Canva;
* Figma AI;
* корпоративные шаблоны документов.

### Для кого

Для всех сотрудников.

### Минимальный уровень

A1–A2.

### Ограничение

Красивый документ не равен качественному управленческому материалу. Содержание, логика, данные и выводы важнее оформления.

## 9.10. Таблицы, данные и BI

### Назначение

Инструменты для работы с цифрами, выгрузками, отчётами и управленческой аналитикой.

Используются для:

* анализа таблиц;
* поиска отклонений;
* подготовки выводов;
* проверки гипотез;
* сводок;
* расчётов;
* визуализации;
* мониторинга KPI;
* подготовки управленческих решений.

### Примеры

* Excel;
* Google Sheets;
* Power BI;
* Looker Studio;
* ChatGPT Data Analysis;
* Python;
* SQL;
* BI-ассистенты;
* корпоративные аналитические витрины.

### Для кого

* руководители;
* аналитики;
* коммерция;
* операционные команды;
* финансы;
* продукт;
* маркетинг;
* закупки;
* контакт-центр.

### Минимальный уровень

A2 для базовой работы.
A3–A4 для создания повторяемых аналитических сценариев.
A5 для перестройки управленческой отчётности.

### Ограничение

ИИ может помочь найти закономерности, но не должен подменять владение данными. Сотрудник обязан понимать, откуда данные, что они означают и где возможны ошибки.

## 9.11. Workflow automation

### Назначение

Workflow automation связывает разные системы, события, данные и действия в цепочки.

Пример:

> Новая заявка → агент классифицирует → система создаёт задачу → ответственный получает уведомление → руководитель видит сводку.

### Примеры инструментов

* n8n;
* Make;
* Zapier;
* Power Automate;
* корпоративные workflow-системы;
* API/webhooks.

n8n поддерживает AI workflows, AI Agent node, инструменты для retriever-сценариев, calculator, custom code tool и MCP Client; это важно для агентной модели, потому что ИИ может быть не отдельным чатом, а шагом внутри процесса. ([n8n Docs][3])

### Для кого

* руководители;
* старшие/лиды;
* ведущие эксперты;
* аналитики;
* операционные команды;
* продукт;
* ИТ;
* сотрудники с A3+.

### Минимальный уровень

A3 для простых сценариев.
A4–A5 для командных и процессных сценариев.

### Ограничение

Workflow без понятной задачи превращается в хрупкую автоматизацию. Сначала нужно описать процесс, данные, риск, владельца и критерии качества.

## 9.12. Agent/workflow builders

### Назначение

Платформы для создания AI-приложений, агентов, сценариев, RAG-пайплайнов и внутренних помощников.

### Примеры

* Dify;
* Langflow;
* Flowise;
* Copilot Studio;
* OpenAI Agents SDK;
* LangGraph;
* CrewAI;
* AutoGen;
* внутренние agent frameworks.

### Для кого

* ИТ;
* продукт;
* аналитики;
* продвинутые руководители;
* владельцы процессов;
* сотрудники A4+.

### Минимальный уровень

A4 для прикладного создания.
A5–A6 для процессного и стратегического применения.

### Ограничение

Не все сотрудники должны создавать агентов с нуля. Большинству достаточно уметь пользоваться готовыми агентами и правильно ставить им задачи.

## 9.13. Low-code / no-code инструменты

### Назначение

Быстрое создание MVP, внутренних форм, таблиц, интерфейсов, баз данных и простых приложений без полноценной разработки.

### Примеры

* Baserow;
* NocoDB;
* Retool;
* Appsmith;
* NocoBase;
* Airtable;
* внутренние low-code платформы.

### Для кого

* руководители;
* аналитики;
* операционные команды;
* продукт;
* проектные менеджеры;
* сотрудники A3+.

### Минимальный уровень

A3.

### Что можно делать

* простые внутренние базы;
* формы заявок;
* управленческие таблицы;
* согласования;
* MVP процессов;
* интерфейсы для ручной проверки;
* панели контроля.

### Ограничение

Low-code не заменяет ИТ в боевых системах, где есть критичные данные, безопасность, интеграции, высокая нагрузка или юридические риски.

---

## 9.14. Coding agents и AI для разработки

### Назначение

Инструменты для ускорения разработки, автоматизации, анализа кода, написания скриптов, тестов и интеграций.

### Примеры

* Cursor;
* GitHub Copilot;
* Claude Code;
* Kilo Code;
* OpenHands;
* Continue;
* Codeium;
* Replit;
* локальные coding-модели.

Claude Code описывается Anthropic как agentic coding tool, работающий в терминале, который понимает кодовую базу и помогает редактировать файлы, запускать команды и управлять Git workflow; это пример перехода от «подсказок по коду» к агентной разработке. ([Anthropic][6])

### Для кого

* разработчики;
* аналитики с Python/SQL;
* ИТ;
* технические лиды;
* продвинутые пользователи A3+;
* руководители, которые хотят собрать технический MVP через агента.

### Минимальный уровень

A3 для простых скриптов и прототипов.
A4–A5 для производственных сценариев.

### Ограничение

Код, созданный агентом, должен проходить ревью, тестирование и проверку безопасности. Нельзя напрямую переносить сгенерированный код в критичные системы без контроля.

### Примеры

* Canva AI;
* Gamma;

* Ideogram;
* DALL·E;
* Runway;
* Adobe Firefly.

### Для кого

* маркетинг;
* продукт;
* HR;

* коммерция;
* руководители;
* дизайнеры;
* сотрудники, готовящие материалы.

### Минимальный уровень

A1–A2 для базового использования.

## 9.16. Контроль, безопасность и governance

### Назначение

Инструменты и правила, которые ограничивают риски при работе с ИИ.

Нужны для:

* защиты данных;
* разграничения доступов;
* аудита действий;
* фильтрации чувствительной информации;
* проверки результата;
* контроля внешних моделей;
* соблюдения требований компании;
* предотвращения утечек.

### Примеры

* DLP;
* access control;
* audit logs;
* model routing;
* guardrails;
* redaction/masking;
* prompt injection checks;
* approval workflow;
* human-in-the-loop;
* policy checks.

OpenAI Agents SDK включает guardrails как механизм проверки входов и выходов агента, а Microsoft Copilot Studio в автономных агентах также использует guardrails наряду с triggers и instructions. ([openai.github.io][5])

### Для кого

* ИТ;
* безопасность;
* владельцы процессов;
* ТОПы;
* руководители;
* сотрудники, работающие с чувствительными данными.

### Минимальный уровень

A2 — понимать базовые запреты.
A4+ — учитывать безопасность при проектировании сценариев.

### Ограничение

Безопасность не должна убить внедрение, но должна задавать контуры: какие данные куда можно передавать, какие действия агент может выполнять сам, а где нужен человек.

## 9.17. Рекомендованная корпоративная карта инструментов

Для практического внедрения лучше разделить инструменты на 4 контура.

## 9.18. Минимальный набор инструментов компании

Чтобы стандарт не остался декларацией, компания должна обеспечить минимум:

| Минимум                             | Зачем нужен                        |
| ----------------------------------- | ---------------------------------- |
| **Корпоративный AI-чат**            | Безопасная точка входа для всех    |
| **Библиотека готовых промптов**     | Быстрый старт без хаоса            |
| **Библиотека AI-сценариев**         | Повторяемое применение             |
| **Несколько готовых агентов**       | Снятие типовой рутины              |
| **Правила работы с данными**        | Безопасность                       |
| **Инструмент workflow automation**  | Переход от чата к процессам        |
| **RAG / база знаний**               | Работа с корпоративной информацией |
| **Метрики использования и эффекта** | Понимание пользы                   |
| **Ответственные владельцы**         | Чтобы инструменты развивались      |

## 9.19. Что не должно происходить

| Ошибка                                             | Почему опасно                                        |
| -------------------------------------------------- | ---------------------------------------------------- |
| Выдать всем доступ к разным ИИ без правил          | Риск утечек и хаоса                                  |
| Ограничиться ChatGPT                               | Не будет agent-first модели                          |
| Купить Copilot и считать трансформацию завершённой | Инструмент без процессов не меняет систему           |
| Начать с тяжёлой автоматизации                     | Можно потратить месяцы без эффекта                   |
| Не дать сотрудникам готовые сценарии               | Большинство останется на бытовом уровне              |
| Не измерять эффект                                 | Будет имитация внедрения                             |
| Отдать всю тему ИТ                                 | ИИ-трансформация — бизнесовая, не только техническая |
| Не подключить безопасность                         | Риски утечек и неправильных действий                 |
| Не назначить владельцев сценариев                  | Всё быстро деградирует                               |

## 9.20. Главный вывод раздела

> Инструменты — это не цель.
> Цель — новая модель выполнения работы: сотрудник ставит задачу, ИИ и агенты снимают рутину, автоматизация связывает действия, человек проверяет качество и отвечает за результат.
