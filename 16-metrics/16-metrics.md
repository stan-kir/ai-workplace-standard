# 16. Метрики эффекта и производительности

Этот раздел фиксирует, как измерять реальный эффект от AI/agent-first подхода.

Главная идея:

> Внедрение ИИ должно оцениваться не по количеству пользователей, промптов и разговоров с чат-ботом, а по изменению производительности: меньше ручной рутины, выше скорость, лучше качество, ниже зависимость от людей и ИТ, выше управляемость процессов.

---

## 16.1. Зачем нужны метрики

Без метрик AI-внедрение быстро превращается в имитацию:

* сотрудники «что-то спрашивают у ИИ»;
* руководители говорят, что команда «использует ChatGPT»;
* появляются красивые кейсы без экономического эффекта;
* автоматизации создаются ради автоматизаций;
* непонятно, стало ли быстрее, дешевле и качественнее.

Метрики нужны, чтобы отделить:

| Имитация                   | Реальный эффект                                 |
| -------------------------- | ----------------------------------------------- |
| «Мы используем ИИ»         | Сократили ручную работу на X часов              |
| «Сделали агента»           | Агент стабильно закрывает Y% типовых задач      |
| «Провели обучение»         | Сотрудники начали применять сценарии в работе   |
| «Создали промпты»          | Промпты используются регулярно и дают результат |
| «Автоматизировали процесс» | Снизили ошибки, время и зависимость от людей    |

---

## 16.2. Принцип измерения

Каждый AI-сценарий должен иметь минимум одну метрику эффекта.

Базовая формула:

> **Было → стало → эффект → риск → вывод.**

| Блок       | Что фиксируем                                 |
| ---------- | --------------------------------------------- |
| **Было**   | Как задача выполнялась раньше                 |
| **Стало**  | Как выполняется после ИИ/агента/автоматизации |
| **Эффект** | Что улучшилось количественно или качественно  |
| **Риск**   | Где ухудшилось или появились ограничения      |
| **Вывод**  | Масштабировать, доработать, остановить        |

---

## 16.3. Основные группы метрик

| Группа                | Что измеряет                                          |
| --------------------- | ----------------------------------------------------- |
| **Скорость**          | Насколько быстрее выполняется задача                  |
| **Ручная работа**     | Сколько человеческого труда снято                     |
| **Качество**          | Стало ли меньше ошибок и доработок                    |
| **Повторяемость**     | Стабильно ли сценарий даёт приемлемый результат       |
| **Самостоятельность** | Меньше ли зависимость от руководителя, ИТ, аналитиков |
| **Масштабирование**   | Используют ли сценарий другие сотрудники/команды      |
| **Экономика**         | Есть ли эффект в деньгах, FTE, стоимости операции     |
| **Безопасность**      | Нет ли нарушений, утечек, критичных ошибок            |
| **Культура**          | Меняется ли поведение сотрудников                     |

---

## 16.4. Метрики скорости

| Метрика                        | Как считать                                              |
| ------------------------------ | -------------------------------------------------------- |
| **Время выполнения задачи**    | Было X минут/часов → стало Y                             |
| **Время подготовки материала** | Сколько занимает отчёт, письмо, сводка, презентация      |
| **Lead time задачи**           | От постановки до готового результата                     |
| **Время реакции**              | Насколько быстрее сотрудник/команда реагирует на запрос  |
| **Время подготовки решения**   | Насколько быстрее появляются варианты и выводы           |
| **Время поиска информации**    | Сколько времени уходит на поиск ответа/регламента/данных |

Пример:

> Подготовка еженедельной сводки занимала 3 часа. После AI-сценария — 50 минут. Эффект: минус 2 часа 10 минут на одну сводку.

---

## 16.5. Метрики снятия ручной работы

| Метрика                                                   | Что показывает                                         |
| --------------------------------------------------------- | ------------------------------------------------------ |
| **Снятые ручные часы**                                    | Сколько часов работы больше не выполняется вручную     |
| **Снижение ручных операций**                              | Сколько шагов процесса убрали или автоматизировали     |
| **Доля задач через AI-сценарии**                          | Какой процент типовых задач выполняется через сценарии |
| **Количество повторяемых задач, переведённых в сценарии** | Сколько рутины упаковано                               |
| **Снижение нагрузки на сотрудников**                      | Меньше ли ручного копирования, сверки, подготовки      |
| **Снижение зависимости от конкретного человека**          | Меньше ли «только он знает, как это делать»            |

Пример:

> В отделе было 12 регулярных ручных отчётов. За квартал 5 переведены в AI-сценарии. Ручная подготовка снизилась на 28 часов в месяц.

---

## 16.6. Метрики качества

| Метрика                                       | Что показывает                                            |
| --------------------------------------------- | --------------------------------------------------------- |
| **Количество ошибок**                         | Стало ли меньше фактических, логических, расчётных ошибок |
| **Количество возвратов на доработку**         | Меньше ли переделок после проверки                        |
| **Полнота результата**                        | Все ли обязательные блоки присутствуют                    |
| **Соответствие шаблону**                      | Выполнен ли стандарт оформления/структуры                 |
| **Качество выводов**                          | Есть ли применимые рекомендации, а не общие слова         |
| **Количество выявленных рисков**              | Помогает ли ИИ находить слабые места раньше               |
| **Доля результатов, принятых с первого раза** | Насколько результат сразу пригоден к работе               |

Пример:

> До AI-чек-листа 40% документов возвращались на доработку. После внедрения — 22%. Эффект: меньше повторной работы и выше качество первичной подготовки.

---

## 16.7. Метрики повторяемости

AI-сценарий ценен, если он работает не один раз, а стабильно.

| Метрика                               | Что показывает                             |
| ------------------------------------- | ------------------------------------------ |
| **Количество запусков сценария**      | Используется ли сценарий реально           |
| **Регулярность использования**        | Используют ли его каждую неделю/день/месяц |
| **Доля успешных запусков**            | Сколько запусков дали применимый результат |
| **Доля ручных исправлений**           | Насколько сильно приходится дорабатывать   |
| **Стабильность качества**             | Не проседает ли результат от раза к разу   |
| **Количество пользователей сценария** | Использует ли его только автор или команда |

Пример:

> Сценарий подготовки протоколов использовался 18 раз за месяц. 15 результатов приняты без существенных доработок. Успешность — 83%.

---

## 16.8. Метрики самостоятельности

Одна из целей стандарта — убрать оправдания «не знаю», «нужен ИТ», «нужен аналитик», «нет времени».

| Метрика                                         | Что показывает                                  |
| ----------------------------------------------- | ----------------------------------------------- |
| **Количество эскалаций без первой попытки**     | Насколько сотрудники пробуют решить задачу сами |
| **Количество задач, решённых без ИТ**           | Где достаточно AI/MVP/low-code                  |
| **Количество задач, решённых без руководителя** | Насколько выросла автономность                  |
| **Доля обращений с подготовленным черновиком**  | Приходят ли сотрудники с первичным результатом  |
| **Скорость выхода на первый вариант**           | Как быстро появляется черновик решения          |
| **Снижение зависимости от отдельных экспертов** | Знания стали доступнее через сценарии/базы      |

Пример:

> Раньше сотрудники приносили руководителю проблему без вариантов. После внедрения стандарта 70% обращений приходят с AI-подготовленным черновиком и 2–3 вариантами решения.

---

## 16.9. Метрики экономики

Экономический эффект не всегда нужно считать идеально, но грубая оценка обязательна.

| Метрика                          | Как считать                                   |
| -------------------------------- | --------------------------------------------- |
| **Экономия часов**               | Снятые часы × частота задачи                  |
| **Экономия FTE**                 | Снятые часы / месячный фонд рабочего времени  |
| **Стоимость операции**           | Было X рублей/операция → стало Y              |
| **Снижение потребности в найме** | Какие задачи закрыты без новых людей          |
| **Снижение стоимости ошибки**    | Меньше переделок, штрафов, потерь, конфликтов |
| **Рост пропускной способности**  | Больше задач на ту же команду                 |
| **Скорость запуска инициатив**   | Больше MVP без привлечения ИТ                 |

Пример:

> Команда снимала 80 часов ручной работы в месяц. Это примерно 0,5 FTE. Если стоимость FTE 200 тыс. руб./мес., потенциальный эффект — около 100 тыс. руб./мес. до учёта стоимости инструментов и проверки качества.

---

## 16.10. Метрики безопасности

AI-сценарий не считается успешным, если он ускорил работу, но создал критичные риски.

| Метрика                                                     | Что показывает                           |
| ----------------------------------------------------------- | ---------------------------------------- |
| **Количество нарушений правил данных**                      | Загружались ли запрещённые данные        |
| **Количество инцидентов**                                   | Были ли ошибки, утечки, опасные действия |
| **Доля сценариев с human-in-the-loop**                      | Есть ли контроль в рискованных задачах   |
| **Наличие владельца сценария**                              | Кто отвечает за сценарий                 |
| **Наличие логирования**                                     | Можно ли восстановить действия           |
| **Количество остановленных рискованных запусков**           | Работает ли контроль                     |
| **Процент сотрудников, прошедших обучение по безопасности** | Базовая зрелость команды                 |

Пример:

> Сценарий не масштабируется, пока не определены: владелец, доступы, данные, проверка человеком, журнал действий и порядок остановки.

---

## 16.11. Метрики культуры и поведения

Эти метрики сложнее, но они показывают, меняется ли модель работы.

| Метрика                                                  | Что показывает                    |
| -------------------------------------------------------- | --------------------------------- |
| **Доля сотрудников, использующих AI-сценарии регулярно** | Проникновение стандарта           |
| **Количество созданных шаблонов и сценариев**            | Активность по снятию рутины       |
| **Количество сценариев, используемых другими**           | Реальная полезность               |
| **Количество предложений по автоматизации**              | Видят ли сотрудники рутину        |
| **Доля задач с AI-первой попыткой**                      | Меняется ли поведение             |
| **Количество обученных сотрудников**                     | Базовое распространение           |
| **Количество AI-амбассадоров**                           | Есть ли внутренняя сеть поддержки |
| **Сопротивление / отказ от применения**                  | Где культурные риски              |

---

## 16.12. Метрики для разных уровней ролей

## Специалисты

| Что измерять                      | Пример                             |
| --------------------------------- | ---------------------------------- |
| Использует ли готовые AI-сценарии | Количество задач через сценарии    |
| Проверяет ли результат            | Доля работ без критичных AI-ошибок |
| Сокращает ли время                | Было/стало по типовым задачам      |
| Соблюдает ли безопасность         | Нет нарушений данных               |

---

## Ведущие эксперты

| Что измерять                         | Пример                              |
| ------------------------------------ | ----------------------------------- |
| Созданы ли экспертные шаблоны        | Количество чек-листов, сценариев    |
| Используются ли другими              | Количество пользователей            |
| Улучшилось ли качество               | Меньше ошибок/возвратов             |
| Снизилась ли зависимость от эксперта | Команда решает типовые вопросы сама |

---

## Старшие / лиды

| Что измерять        | Пример                                  |
| ------------------- | --------------------------------------- |
| Сценарии для группы | Количество внедрённых сценариев         |
| Обучение коллег     | Сколько сотрудников используют сценарий |
| Снятая рутина       | Часы/операции в зоне лида               |
| Качество результата | Меньше доработок                        |

---

## Руководители

| Что измерять         | Пример                                  |
| -------------------- | --------------------------------------- |
| Карта ручной рутины  | Есть / нет / обновляется                |
| AI-сценарии отдела   | Количество запущенных и работающих      |
| Эффект               | Часы, FTE, скорость, ошибки             |
| Применение командой  | Доля сотрудников, использующих сценарии |
| Управление качеством | Есть чек-листы, контроль, владельцы     |

---

## ТОПы

| Что измерять                  | Пример                                          |
| ----------------------------- | ----------------------------------------------- |
| Портфель AI-инициатив функции | Количество приоритетных процессов               |
| Экономический эффект          | FTE, стоимость операций, скорость решений       |
| Изменение ролей               | Обновлены требования и грейды                   |
| Масштабирование               | Сценарии внедрены не точечно, а системно        |
| Governance                    | Есть правила, владельцы, контроль, безопасность |

---

## 16.13. Минимальная карточка метрики AI-сценария

Для каждого сценария можно использовать простую карточку.

## Название сценария

Что делает сценарий.

## Владелец

Кто отвечает за работу и качество.

## Проблема

Какая ручная работа или ограничение были.

## Метрика baseline

Как было до внедрения.

## Целевой эффект

Что хотим улучшить.

## Фактический эффект

Что получили.

## Риски

Что может пойти не так.

## Решение

Масштабировать / доработать / остановить.

---

## 16.14. Пример карточки

## Название

AI-подготовка еженедельной сводки по задачам отдела.

## Владелец

Руководитель отдела.

## Проблема

Сводка готовилась вручную, занимала 3 часа каждую неделю.

## Метрика baseline

3 часа × 4 раза в месяц = 12 часов ручной работы в месяц.

## Целевой эффект

Сократить время минимум на 50% без потери качества.

## Фактический эффект

Время сократилось до 1 часа на сводку. Экономия — 8 часов в месяц.

## Риски

ИИ может неверно интерпретировать статусы задач.

## Контроль

Руководитель проверяет выводы перед планёркой.

## Решение

Оставить сценарий, доработать шаблон и протестировать автоматическую выгрузку.

---

## 16.15. Что не стоит измерять как главные KPI

Не надо делать основными KPI:

| Метрика                            | Почему слабая                      |
| ---------------------------------- | ---------------------------------- |
| Количество запросов к ИИ           | Много запросов не равно польза     |
| Количество пользователей           | Люди могли попробовать один раз    |
| Количество созданных промптов      | Промпты могут не использоваться    |
| Количество обученных сотрудников   | Обучение не равно изменение работы |
| Количество внедрённых инструментов | Инструмент не равен результату     |
| Количество агентов                 | Агенты могут быть бесполезными     |
| Потраченные токены                 | Расход не равен эффекту            |

Эти метрики можно смотреть как технические или вспомогательные, но не как итоговую оценку пользы.

---

## 16.16. Правильные итоговые KPI

Лучше смотреть:

| KPI                                           | Почему полезен                       |
| --------------------------------------------- | ------------------------------------ |
| **Снятые часы ручной работы**                 | Прямой показатель производительности |
| **Снижение времени выполнения задач**         | Показывает скорость                  |
| **Снижение ошибок / возвратов**               | Показывает качество                  |
| **Количество реально используемых сценариев** | Показывает внедрение                 |
| **Доля команды, работающей через сценарии**   | Показывает изменение поведения       |
| **Экономия FTE / стоимости операции**         | Показывает экономику                 |
| **Количество процессов с AI-пересборкой**     | Показывает зрелость                  |
| **Снижение зависимости от ИТ/экспертов**      | Показывает автономность              |
| **Отсутствие критичных инцидентов**           | Показывает управляемую безопасность  |

---

## 16.17. Уровни зрелости измерения

| Уровень              | Как измеряем                                     |
| -------------------- | ------------------------------------------------ |
| **0. Не измеряем**   | Есть разговоры про ИИ, эффекта не видно          |
| **1. Активность**    | Смотрим пользователей, запросы, обучение         |
| **2. Применение**    | Смотрим задачи, сценарии, регулярность           |
| **3. Эффект**        | Смотрим время, ошибки, ручные часы               |
| **4. Экономика**     | Считаем FTE, стоимость операции, ROI             |
| **5. Трансформация** | Видим изменение процессов, ролей и модели работы |

Норма для компании — двигаться минимум к уровню 3, а для руководителей и ТОПов — к уровням 4–5.

---

## 16.18. Частота измерения

| Уровень                | Частота                       |
| ---------------------- | ----------------------------- |
| **AI-сценарий**        | После теста, затем ежемесячно |
| **Команда / отдел**    | Ежемесячно                    |
| **Функция**            | Ежеквартально                 |
| **Компания**           | Ежеквартально / раз в полгода |
| **Performance review** | По циклу оценки сотрудников   |

---

## 16.19. Ответственность за метрики

| Кто                   | За что отвечает                            |
| --------------------- | ------------------------------------------ |
| **Сотрудник**         | Фиксирует применение в своих задачах       |
| **Лид / старший**     | Фиксирует сценарии в своей зоне            |
| **Руководитель**      | Считает эффект по команде                  |
| **ТОП**               | Смотрит эффект по функции                  |
| **HR / HRBP**         | Встраивает в оценку, обучение, развитие    |
| **ИТ / AI-платформа** | Даёт техническую статистику, доступы, логи |
| **Безопасность**      | Отслеживает риски и инциденты              |

---

## 16.20. Главный вывод раздела

> AI/agent-first подход считается внедрённым не тогда, когда сотрудники получили доступ к ИИ, а когда изменилась производительность: задачи выполняются быстрее, ручной рутины меньше, качество выше, сотрудники самостоятельнее, процессы управляемее, а риски остаются под контролем.
