# 19. Порядок внедрения стандарта

Этот раздел фиксирует, как внедрять AI/agent-first стандарт в компании, чтобы он не остался документом, обучающим курсом или лозунгом.

Главная идея:

> Сначала фиксируем норму, потом даём инструменты, затем обучаем, внедряем в процессы управления людьми и только после этого требуем зрелого результата.

---

## 19.1. Главный принцип внедрения

Нельзя внедрять стандарт только через рассылку документа.

Неправильно:

> Разослали стандарт → провели обучение → считаем, что компания стала AI-first.

Правильно:

> Стандарт → инструменты → безопасность → обучение → практика → оценка → метрики → управленческое требование → постоянное улучшение.

---

## 19.2. Этапы внедрения

| Этап                              | Суть                                              | Результат                              |
| --------------------------------- | ------------------------------------------------- | -------------------------------------- |
| **1. Утверждение стандарта**      | Зафиксировать новую норму работы                  | Есть официальный документ              |
| **2. Карта ролей и уровней**      | Определить минимальный AI-уровень для каждой роли | Есть требования по ролям               |
| **3. Инструменты и безопасность** | Дать сотрудникам безопасную среду                 | Есть разрешённые инструменты и правила |
| **4. Обучение**                   | Вывести сотрудников на базовый уровень            | Есть минимальная AI-грамотность        |
| **5. Первые сценарии**            | Запустить реальные AI-сценарии в командах         | Есть практический эффект               |
| **6. Встраивание в HR-процессы**  | Найм, адаптация, оценка, грейдинг                 | Стандарт стал частью управления людьми |
| **7. Метрики эффекта**            | Измерять пользу, а не активность                  | Видно, что изменилось                  |
| **8. Масштабирование**            | Распространять лучшие сценарии                    | Agent-first становится системой        |

---

## 19.3. Этап 1. Утверждение стандарта

## Цель

Сделать стандарт официальной управленческой нормой.

## Что нужно сделать

1. Утвердить название документа.
2. Утвердить главный принцип.
3. Утвердить уровни зрелости A1–A6.
4. Утвердить требования по ролям.
5. Утвердить обязательность стандарта для всех сотрудников.
6. Назначить владельцев внедрения.
7. Определить дату вступления стандарта в силу.

## Важная формулировка

> Стандарт вводится не как дополнительная нагрузка, а как новая норма производительности: меньше ручной рутины, больше самостоятельности, выше скорость и качество результата.

---

## 19.4. Этап 2. Карта ролей и минимальных уровней

## Цель

Связать стандарт с реальными ролями компании.

## Что нужно сделать

Для каждой роли определить:

* текущий уровень ожиданий;
* минимальный AI-уровень;
* целевой AI-уровень;
* обязательные AI-сценарии роли;
* обучение;
* критерии оценки;
* срок выхода на уровень.

## Пример

| Роль                | Минимум | Цель | Срок выхода  |
| ------------------- | ------: | ---: | ------------ |
| Специалист КЦ       |      A2 |   A3 | 60 дней      |
| Руководитель КЦ     |      A4 |   A5 | 90 дней      |
| Аналитик            |      A3 |   A4 | 90 дней      |
| Product Manager     |      A4 |   A5 | 90 дней      |
| Руководитель отдела |      A4 |   A5 | 90–120 дней  |
| ТОП                 |      A5 |   A6 | 120–180 дней |

---

## 19.5. Адаптация под размер компании

Подход к внедрению зависит от масштаба организации. Ниже — рекомендации по адаптации стандарта.

### Малая компания (до 50 сотрудников)

* Пропустить этапы формализации: утверждение стандарта = решение CEO на совещании.
* Не нужна карта ролей — достаточно определить 2–3 ключевые роли.
* Обучение = 2–3 практических воркшопа для всех.
* Первые сценарии = CEO / основатели показывают на своём примере.
* Метрики = простые (часы, качество), без сложных дашбордов.
* AI-амбассадор = 1 человек (часто CTO или COO).
* Из приложений взять: Приложение 3 (шаблон запроса), Приложение 4 (чек-лист проверки), Приложение 7 (карта рутины).

### Средняя компания (50–500 сотрудников)

* Стандарт можно внедрять по отделам.
* Пилот = 2–3 команды, потом масштабирование.
* Нужен 1 владелец внедрения на полставки.
* Карта ролей — для ключевых функций (продажи, маркетинг, операции).
* Сеть AI-амбассадоров = 3–5 человек.
* Из приложений дополнительно: Приложение 5 (AI-сценарий), Приложение 8 (MVP), Приложение 9 (оценка).

### Крупная компания (500+ сотрудников)

* Полный цикл по разделу 19.
* Выделенная роль или команда внедрения.
* Централизованная AI-платформа.
* Формальная интеграция с HR-системами.
* Governance board для AI-инициатив.
* Ежеквартальный отчёт по функциям.
* Использовать все 24 приложения.

---

## 19.6. Этап 3. Инструменты и безопасный контур

## Цель

Сотрудники не должны уходить в хаотичное использование внешних сервисов.

## Минимум, который должна дать компания

| Что                           | Зачем                                 |
| ----------------------------- | ------------------------------------- |
| **Корпоративный AI-чат**      | Безопасная точка входа                |
| **Разрешённые инструменты**   | Понятно, чем можно пользоваться       |
| **Правила данных**            | Понятно, что можно и нельзя загружать |
| **Шаблоны запросов**          | Быстрый старт                         |
| **Библиотека сценариев**      | Повторяемое применение                |
| **Несколько готовых агентов** | Быстрая польза                        |
| **Канал поддержки**           | Куда идти с вопросами                 |
| **Ответственные владельцы**   | Кто развивает систему                 |

## Важный риск

Если компания не даст удобный безопасный контур, сотрудники всё равно будут использовать внешние AI-сервисы, но скрытно и без контроля.

---

## 19.7. Этап 4. Обучение сотрудников

## Цель

Вывести всех сотрудников на минимальный уровень своей роли.

## Порядок

1. Базовый курс для всех.
2. Отдельный курс по безопасности.
3. Практика на реальных задачах.
4. Ролевые треки.
5. Обучение руководителей отдельно.
6. Обучение AI-амбассадоров.
7. Проверка результата через рабочие задания.

## Что не делать

Не проводить длинное теоретическое обучение без практики.

Итог обучения должен быть не сертификат, а рабочий результат:

* шаблон;
* сценарий;
* чек-лист;
* агент;
* автоматизированная сводка;
* карта рутины;
* улучшенный процесс.

---

## 19.8. Этап 5. Первые AI-сценарии в командах

## Цель

Быстро показать, что стандарт даёт практический эффект.

## Как выбирать первые сценарии

Выбирать не самые сложные и не самые эффектные, а те, где есть:

* регулярная ручная работа;
* понятные входные данные;
* низкий или средний риск;
* быстрый MVP;
* понятный владелец;
* измеримый эффект;
* возможность масштабирования.

## Хорошие первые сценарии

| Сценарий                      | Почему подходит               |
| ----------------------------- | ----------------------------- |
| Протоколы встреч              | Быстрый эффект, низкий риск   |
| Еженедельные сводки           | Регулярность, измеримое время |
| Анализ просрочек задач        | Управленческая польза         |
| Резюме документов             | Частая рутина                 |
| Проверка текстов и материалов | Качество и скорость           |
| Классификация обращений       | Операционная польза           |
| Подготовка FAQ                | База для RAG и поддержки      |
| Черновики инструкций          | Быстрая стандартизация        |
| Анализ отзывов / обращений    | Полезно продукту и КЦ         |

## Что не выбирать первым

* критичные финансовые операции;
* автоматические действия с клиентами;
* юридически значимые ответы;
* изменение цен;
* действия с персональными данными;
* сложные интеграции без владельца;
* проекты, где сначала нужен порядок в данных.

---

## 19.9. Этап 6. Встраивание в найм

## Что изменить

1. Добавить AI/agent-first блок во все вакансии.
2. Добавить вопросы на интервью.
3. Добавить тестовые задания.
4. Ввести минимальные уровни по ролям.
5. Фиксировать AI-зрелость в интервью-карте.
6. Не нанимать руководителей, которые не готовы внедрять agent-first подход.

## Главная проверка

> Кандидат умеет работать в новой производительной среде или будет продолжать старую ручную модель?

---

## 19.10. Этап 7. Встраивание в адаптацию

## Что изменить

1. Включить стандарт в первый день адаптации.
2. Дать доступ к AI-инструментам.
3. Провести инструктаж по безопасности.
4. Дать первые задания через ИИ.
5. Проверять AI-зрелость на 30/60/90 дней.
6. Фиксировать сопротивление или прогресс.

## Минимум к 90-му дню

Сотрудник должен подтвердить соответствие минимальному AI-уровню роли или иметь понятный план доразвития.

---

## 19.11. Этап 8. Встраивание в оценку и performance review

## Что добавить

В performance review должен появиться отдельный блок:

> **AI/agent-first вклад сотрудника.**

Проверять:

* какие задачи ускорил;
* какие сценарии использовал;
* какие шаблоны создал;
* что перестал делать вручную;
* какой эффект получил;
* как проверял качество;
* какие риски учёл;
* что предлагает улучшить дальше.

---

## 19.12. Этап 9. Встраивание в грейдинг и карьерный рост

## Главная норма

> Нельзя расти в роли, если сотрудник остаётся в старой ручной модели работы.

Что нужно сделать:

1. Добавить AI-уровни в требования к грейдам.
2. Зафиксировать минимальный уровень для повышения.
3. Связать AI-зрелость с кадровым резервом.
4. Не повышать руководителей, которые не внедряют AI-сценарии в команде.
5. Не повышать экспертов, которые не масштабируют экспертизу через инструменты.

---

## 19.13. Этап 10. Метрики и регулярный контроль

## Минимальный набор метрик

| Метрика                                       | Что показывает             |
| --------------------------------------------- | -------------------------- |
| Количество работающих AI-сценариев            | Есть ли реальное внедрение |
| Снятые ручные часы                            | Производственный эффект    |
| Сокращение времени задач                      | Скорость                   |
| Снижение ошибок / возвратов                   | Качество                   |
| Доля сотрудников, использующих сценарии       | Проникновение              |
| Количество сценариев, созданных командами     | Активность снизу           |
| Количество нарушений безопасности             | Контроль рисков            |
| Количество процессов в agent-first пересборке | Зрелость трансформации     |

---

## 19.14. Управленческий ритм внедрения

## Еженедельно

На уровне команд:

* какие сценарии тестируются;
* какие задачи удалось ускорить;
* где возникли ошибки;
* что нужно доработать.

## Ежемесячно

На уровне отдела:

* карта ручной рутины;
* новые сценарии;
* снятые часы;
* качество;
* сопротивление;
* обучение.

## Ежеквартально

На уровне функции / компании:

* эффект по функциям;
* лучшие сценарии;
* риски;
* изменения ролей;
* приоритеты следующего квартала;
* решения по масштабированию.

---

## 19.15. Роли во внедрении

| Участник              | Роль во внедрении                                    |
| --------------------- | ---------------------------------------------------- |
| **CEO / ТОПы**        | Утверждают норму, требуют внедрения, убирают барьеры |
| **Руководители**      | Внедряют в командах, считают эффект                  |
| **HR / HRBP**         | Встраивают в найм, адаптацию, обучение, оценку       |
| **ИТ / AI-платформа** | Даёт инструменты, доступы, интеграции, поддержку     |
| **ИБ / юристы**       | Задают правила безопасности                          |
| **AI-амбассадоры**    | Помогают командам запускать сценарии                 |
| **Сотрудники**        | Применяют, проверяют, предлагают улучшения           |

---

## 19.16. Первые 30 дней внедрения

## Цель

Запустить стандарт без перегруза и получить первые доказательства пользы.

## Что сделать

1. Утвердить стандарт.
2. Назначить владельца внедрения.
3. Утвердить минимальные уровни по ролям.
4. Определить разрешённые инструменты.
5. Утвердить правила безопасности.
6. Провести короткий вводный брифинг для руководителей.
7. Выбрать 3–5 пилотных команд.
8. В каждой пилотной команде выбрать 2–3 ручные рутины.
9. Запустить первые AI-сценарии.
10. Зафиксировать baseline: сколько времени занимало раньше.

## Результат 30 дней

* стандарт утверждён;
* пилоты выбраны;
* первые сценарии запущены;
* есть baseline;
* есть первые ошибки и выводы.

---

## 19.17. Первые 60 дней

## Цель

Перейти от первых проб к регулярному применению.

## Что сделать

1. Провести базовое обучение для пилотных команд.
2. Запустить библиотеку промптов и сценариев.
3. Собрать обратную связь.
4. Улучшить первые сценарии.
5. Подключить руководителей к ежемесячной карте рутины.
6. Встроить AI-блок в адаптацию.
7. Обновить шаблоны вакансий.
8. Начать оценивать эффект.

## Результат 60 дней

* есть первые работающие сценарии;
* сотрудники используют их регулярно;
* руководители видят эффект;
* понятны ограничения;
* стандарт встроен в найм и адаптацию.

---

## 19.18. Первые 90 дней

## Цель

Сделать стандарт частью управленческой системы.

## Что сделать

1. Расширить внедрение на большее число команд.
2. Включить AI/agent-first блок в performance review.
3. Утвердить связь с грейдами.
4. Сформировать сеть AI-амбассадоров.
5. Выделить лучшие сценарии для масштабирования.
6. Определить процессы для agent-first пересборки.
7. Подготовить квартальный отчёт по эффекту.
8. Принять решения: что масштабировать, что остановить, что доработать.

## Результат 90 дней

* стандарт работает не только как документ;
* есть сценарии, метрики, владельцы;
* руководители включены;
* HR-процессы обновлены;
* понятны следующие приоритеты.

---

## 19.19. Что нельзя делать при внедрении

| Ошибка                            | Почему опасно                                      |
| --------------------------------- | -------------------------------------------------- |
| Начать с большой автоматизации    | Долго, дорого, высокий риск без проверки пользы    |
| Свести всё к обучению ChatGPT     | Не будет agent-first модели                        |
| Не дать безопасные инструменты    | Люди уйдут во внешние сервисы                      |
| Не подключить руководителей       | Команды вернутся к ручной работе                   |
| Не измерять эффект                | Появится имитация                                  |
| Не назначить владельцев сценариев | Всё быстро деградирует                             |
| Сделать стандарт добровольным     | Он не станет нормой                                |
| Наказывать до обучения            | Вызовет сопротивление                              |
| Привязать сразу к премии          | Начнётся имитация и накрутка                       |
| Делать всё через ИТ               | Бизнес не научится сам видеть и запускать сценарии |

---

## 19.20. Правильная последовательность требований

Сотрудников нельзя сразу жёстко оценивать по стандарту, если компания не дала условий.

Правильная последовательность:

1. Объяснили норму.
2. Дали инструменты.
3. Дали правила безопасности.
4. Дали шаблоны.
5. Провели обучение.
6. Дали практические задачи.
7. Проверили первые результаты.
8. Дали обратную связь.
9. Встроили в оценку.
10. Начали требовать как обязательный стандарт.

---

## 19.21. Условия успешного внедрения

Стандарт заработает, если есть:

* ясная позиция CEO / ТОПов;
* обязательность для руководителей;
* безопасный корпоративный AI-контур;
* простые правила;
* практическое обучение;
* первые быстрые сценарии;
* метрики эффекта;
* владельцы сценариев;
* связь с наймом, адаптацией, оценкой и грейдингом;
* регулярный управленческий контроль.

---

## 19.22. Типичные ошибки внедрения: реальные сценарии

Документ описывает целевое состояние. Но на практике внедрение AI-стандарта регулярно сталкивается с типовыми провалами. Их полезно знать заранее.

| Сценарий | Что пошло не так | Урок |
| --- | --- | --- |
| Маркетинг начал генерировать все тексты через ChatGPT | Качество контента упало, появился «AI-слог», аудитория заметила | AI-черновик ≠ финальный продукт; нужен стандарт доработки |
| Руководитель потребовал от всех «использовать ИИ» без инструментов и обучения | Сотрудники массово ушли во внешние бесплатные сервисы, загрузили клиентские данные | Сначала контур и правила, потом требование |
| Агент для ответов клиентам запущен без human-in-the-loop | Агент отправил некорректный ответ с ложной гарантией, клиент потребовал компенсацию | Критичные внешние действия — только с проверкой человеком |
| Команда создала 20 AI-сценариев за месяц | Через 2 месяца никто ни одним не пользовался — не было владельцев и поддержки | Количество сценариев ≠ внедрение; нужен владелец каждого |
| ТОП запустил «AI-трансформацию» с презентации и серии обучений | Через квартал — ноль измеримого эффекта, нарастающий цинизм команды | Начинать с конкретной проблемы и MVP, а не с презентации |
| Аналитик использовал ИИ для финмодели без проверки формул | Ошибка в расчёте маржинальности привела к неверному управленческому решению | ИИ ошибается в математике; расчёты проверяются отдельно |
| Сотрудник загрузил полную клиентскую базу в бесплатный AI-сервис | Данные попали в обучающий датасет модели; риск утечки | Без корпоративного контура люди используют то, что доступно |
| HR добавил AI-зрелость в оценку, не дав обучения и инструментов | Сотрудники восприняли это как наказание за отсутствие навыков, которым их не учили | Последовательность: обучение → практика → оценка |
